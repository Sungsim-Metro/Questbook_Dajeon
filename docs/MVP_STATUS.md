# 모험가의 수첩 MVP 구현 현황

이 문서는 현재 저장소에 실제로 구현되어 있는 범위와 다음 구현 작업을 추적한다. 목표 아키텍처, 데이터 정책, 네트워크 구조, 확장 설계는 `PROJECT_DESIGN.md`에서 관리한다.

## 1. 현재 구현 요약

현재 저장소는 제출용 서비스 흐름을 보여주는 정적 모바일 웹 프로토타입이다. 실제 사용자 계정, 한국관광공사 OpenAPI 추천 데이터, 데이터베이스 저장, Gemini 기반 퀘스트 생성, 실제 퀘스트 인증은 아직 연결되어 있지 않다.

현재 구현은 다음 요소로 구성된다.

- 정적 HTML 화면
- 공통 CSS 스타일
- 목업 데이터 기반 화면 렌더링
- NAVER Dynamic Map 연동
- NAVER Geocoding 및 Reverse Geocoding 서버 프록시
- 지도 API 설정이 없거나 실패할 때 사용하는 목업 지도 fallback

## 2. 구현된 화면

### 2.1 홈 화면

- 파일: `index.html`
- 사용자 레벨, XP, 현재 위치 후보, 획득 뱃지, 주변 퀘스트 수, 보상 수를 표시한다.
- 오늘의 추천 퀘스트를 목업 데이터 기반으로 표시한다.

### 2.2 지도 화면

- 파일: `map.html`
- NAVER Dynamic Map을 사용해 현재 위치와 퀘스트 장소 마커를 표시한다.
- 브라우저 위치 권한을 사용해 현재 위치를 확인할 수 있다.
- 주소 검색과 좌표 입력을 통해 지도 중심을 이동할 수 있다.
- NAVER Maps 설정이 없거나 SDK 로드에 실패하면 목업 지도 UI로 fallback한다.

### 2.3 퀘스트 화면

- 파일: `quests.html`
- 방문형, 이동형, 소비형, 테마형, 활동형 퀘스트를 필터링해 보여준다.
- 현재 데이터는 `assets/js/mock-data.js`의 목업 퀘스트 배열을 사용한다.

### 2.4 탐험 노트 화면

- 파일: `notes.html`
- 완료한 퀘스트와 활동 기록을 수첩 형태로 보여준다.
- 현재 데이터는 목업 활동 기록을 사용한다.

### 2.5 뱃지 화면

- 파일: `badges.html`
- 획득한 뱃지와 잠긴 뱃지, 수첩 스탬프를 표시한다.
- 현재 데이터는 목업 뱃지 정의와 사용자 획득 상태를 사용한다.

## 3. 구현된 서버 기능

현재 `server.py`는 Python 표준 라이브러리 기반 로컬 개발 서버이다. 운영용 웹 서버와 앱 서버 분리는 아직 구현되어 있지 않다.

구현된 기능은 다음과 같다.

- 정적 HTML, CSS, JavaScript, 이미지 파일 제공
- `/api/naver-map/config`에서 NAVER Maps 설정 상태 제공
- `/api/naver-map/status`에서 NAVER Maps 설정 상태 제공
- `/api/naver-map/geocode`에서 NAVER Geocoding API 프록시
- `/api/naver-map/reverse-geocode`에서 NAVER Reverse Geocoding API 프록시
- 브라우저에는 NAVER Maps API Key ID만 전달
- NAVER REST API Key는 서버에서만 사용

## 5. 현재 데이터 상태

현재 화면 데이터는 모두 `assets/js/mock-data.js`에서 제공한다.

- 목업 사용자: 닉네임, 레벨, XP, 완료 퀘스트 수, 보상 수
- 목업 위치: 대전 중앙로 좌표
- 목업 뱃지: 자연, 과학, 원도심, 지역 상권, 이동형, 야경 기록
- 목업 장소: 한밭수목원, 국립중앙과학관, 은행동 스카이로드, 성심당 본점, 타슈 중앙로 거점, 보문산 전망대
- 목업 퀘스트: 방문형, 이동형, 소비형, 테마형, 활동형
- 목업 수첩 기록: 완료 기록과 추천 대기 기록

아직 실제 사용자별 영구 저장소는 없다. 퀘스트 완료, 뱃지, 레벨, 수첩 기록도 현재는 브라우저 화면에 표시되는 정적 목업 상태이다.

## 6. 현재 미구현 범위

다음 기능은 설계되어 있으나 아직 구현되어 있지 않다.

- 한국관광공사 OpenAPI 실제 호출
- 사용자 로그인과 회원 식별
- 사용자 선호도 입력과 저장
- 유저 단위 `TourPlaceCache`
- 추천 점수 계산
- 템플릿 기반 퀘스트 생성
- Gemini 기반 퀘스트 후보 생성
- `ReusableQuest` 공용 퀘스트 저장
- `UserQuestInstance` 사용자별 퀘스트 상태 저장
- GPS 기반 퀘스트 완료 인증
- 사진 인증과 영수증 인증
- XP, 레벨, 뱃지 진행도 갱신
- 모험가의 수첩 영구 저장
- 지역 상권 리워드 발급
- 파트너 포털과 쿠폰 사용 처리
- 운영용 웹 서버와 앱 서버 분리 배포
- 관계형 데이터베이스와 서버 캐시 연동

## 7. 다음 구현 작업

### 7.1 MVP 안정화

- 정적 화면의 모바일 레이아웃과 내비게이션을 검수한다.
- NAVER Maps 설정이 없을 때 fallback UI가 자연스럽게 동작하는지 확인한다.
- `.env.example`과 `README.md`의 로컬 실행 안내가 실제 동작과 일치하는지 확인한다.
- 현재 목업 데이터가 목표 데이터 모델과 지나치게 어긋나지 않도록 필드명을 정리한다.

### 7.2 OpenAPI 추천 기능 연결

- 한국관광공사 OpenAPI 호출 모듈을 추가한다.
- 사용자의 현재 좌표, 관광 카테고리, 거리 조건을 API 요청으로 전달한다.
- OpenAPI 응답에서 추천 계산에 필요한 최소 필드만 추출한다.
- 추출된 관광지 후보를 유저, 위치 권역, 카테고리 기준으로 30분 임시 캐시에 저장한다.
- 목업 퀘스트 목록과 지도 마커를 실제 추천 후보 기반으로 교체한다.

### 7.3 웹 서버와 앱 서버 분리

- 정적 파일 제공과 API 프록시를 담당하는 웹 서버 구성을 준비한다.
- 추천, 인증, 퀘스트, 리워드 로직을 담당하는 Python 앱 서버를 분리한다.
- 로컬 개발 환경에서는 단순 실행을 유지하되, 배포 설계에서는 `웹 서버 -> 앱 서버` 흐름을 따른다.
- 앱 서버만 외부 API와 DB 및 캐시에 접근하도록 네트워크 경계를 정리한다.

### 7.4 사용자 기록과 보상 시스템

- 로그인 또는 간단한 사용자 식별 기능을 추가한다.
- 관계형 DB에 사용자, 선호도, 완료 퀘스트, XP, 레벨, 뱃지 진행도를 저장한다.
- 퀘스트 완료 시 `QuestCompletion`, `LevelProgress`, `Badge`, `AdventureNote` 갱신을 하나의 트랜잭션으로 처리한다.
- 수첩 화면을 실제 완료 이력 기반으로 전환한다.

### 7.5 퀘스트 생성과 재사용

- 관광지 카테고리별 퀘스트 템플릿을 정의한다.
- OpenAPI 관광지 후보에 템플릿을 적용해 퀘스트 후보를 생성한다.
- 생성 결과의 안전성, 인증 방식, XP 보상 범위를 검증한다.
- 검증된 퀘스트는 `ReusableQuest` 공용 자산으로 저장한다.
- 특정 사용자에게 추천되거나 선택된 퀘스트는 `UserQuestInstance`로 관리한다.

### 7.6 Gemini 연동

- Gemini는 퀘스트 문구 다양화와 추가 후보 생성을 위한 보조 수단으로 사용한다.
- AI 생성 결과를 그대로 노출하지 않고 검증 정책을 통과한 결과만 저장한다.
- Gemini 장애 시 템플릿 기반 퀘스트만 제공하는 fallback을 유지한다.

### 7.7 지역 상권 리워드 연계

- `PartnerStore`, `PartnerAccount`, `RewardOffer`, `UserReward`, `RewardRedemption` 모델을 구현한다.
- 퀘스트 완료, 뱃지 획득, 레벨 상승 조건에 따라 리워드를 발급한다.
- 사용자 앱에서 QR 코드 또는 사용 코드를 표시한다.
- 파트너 포털에서 쿠폰 검증과 사용 처리를 할 수 있게 한다.
- 중복 사용 방지는 관계형 DB 트랜잭션으로 처리한다.
- 파트너 화면에는 쿠폰 사용에 필요한 최소 정보만 노출한다.

## 8. 검수 기준

- 로컬 실행 명령으로 정적 화면과 지도 페이지가 열린다.
- NAVER Maps API Key가 없을 때 목업 지도 fallback이 동작한다.
- NAVER REST API Key가 브라우저에 노출되지 않는다.
- 현재 구현 현황은 이 문서에 반영하고, 목표 설계 변경은 `PROJECT_DESIGN.md`에 반영한다.
- 새 기능을 구현할 때는 목업 데이터 의존 범위가 줄어드는지 확인한다.
