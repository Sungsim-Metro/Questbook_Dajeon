# 모험가의 수첩 프로젝트 설계서

## 1. 프로젝트 개요

`모험가의 수첩`은 대전 지역 관광지를 사용자의 현재 위치, 선호도, 이전 퀘스트 수행 이력과 연결해 맞춤형 관광 퀘스트를 제공하는 모바일 웹 기반 관광 게임 서비스이다. 사용자는 주변 관광지를 탐색하고, 장소별 퀘스트를 수행하며, 완료 기록을 뱃지와 레벨, 개인 수첩 형태로 축적한다.

이 프로젝트는 프로젝트 과목 및 공모전 제출을 위한 프로토타입과 확장 설계를 함께 다룬다. 이 문서는 목표 기능, 데이터 정책, 시스템 아키텍처, 네트워크 구조, 보안 원칙, 확장 방향을 정의한다. 현재 저장소에 실제로 구현된 범위와 다음 구현 작업은 `MVP_STATUS.md`에서 관리한다.

### 1.1 문제 정의

대전 관광은 장소 정보 자체보다 사용자가 지금 어디에 있고, 어떤 활동을 선호하며, 다음에 어떤 장소로 이동하면 좋은지 연결해주는 경험이 중요하다. 일반 관광 정보 서비스는 장소 목록을 제공하는 데 그치는 경우가 많아 관광 행동을 지속시키는 동기가 약하다.

`모험가의 수첩`은 관광지를 게임형 퀘스트로 재구성해 다음 문제를 해결한다.

- 현재 위치 주변의 관광지를 사용자의 취향과 이력에 맞게 추천한다.
- 관광지 방문을 퀘스트, XP, 뱃지, 수첩 기록으로 연결해 재방문 동기를 만든다.
- 대전의 자연, 과학, 원도심, 지역 상권, 이동 수단 등 지역 특성을 퀘스트 테마로 표현한다.
- 관광객의 기록을 개인화된 콘텐츠로 남겨 이후 추천과 공유에 활용한다.

### 1.2 목표 사용자

주요 사용자는 대전 지역을 방문하는 관광객, 대전 내 체험형 관광을 원하는 시민, 프로젝트 과목 또는 공모전 평가자이다. MVP에서는 실제 대규모 운영보다 기능 흐름과 서비스 가능성을 보여주는 데 집중한다.

### 1.3 운영 규모 기준

설계상 운영 규모는 전체 사용자 1,000명 이하, 동시 접속 50명 이하로 둔다. 이 수치는 실제 초기 사용자 예측치가 아니라 프로젝트 과목 및 공모전 제출을 위해 넉넉하게 설정한 상한 기준이다. 따라서 초기 인프라는 복잡한 분산 시스템보다 단순한 웹 서버, 앱 서버, 관계형 데이터베이스, 외부 API 연동 구조를 우선한다.

## 2. 문서 범위와 구현 현황 관리

이 문서는 앞으로 구현할 목표 시스템의 설계 기준을 정의한다. 현재 저장소의 실제 구현 상태, 파일 구조, 목업 데이터 사용 범위, 미구현 기능, 다음 작업 목록은 `MVP_STATUS.md`에서 별도로 관리한다.

문서를 분리하는 이유는 다음과 같다.

- 목표 설계 문서가 현재 임시 구현 세부사항에 묶이지 않게 한다.
- 코드 변경으로 자주 바뀌는 구현 현황을 별도 문서에서 빠르게 갱신한다.
- 공모전 또는 과목 제출 시 목표 설계와 구현 진행 상황을 구분해 설명할 수 있게 한다.

## 3. 핵심 사용자 흐름

### 3.1 관광지 추천 흐름

1. 사용자가 모바일 웹에서 현재 위치 사용을 허용한다.
2. 서비스는 현재 좌표, 선택한 관광 카테고리, 사용자의 선호도를 확인한다.
3. 한국관광공사 OpenAPI를 호출해 현재 위치 주변 관광지 후보를 조회한다.
4. 조회 결과를 유저 단위 임시 캐시에 저장한다.
5. 사용자의 완료 퀘스트, 뱃지 이력, 선호 테마, 거리, 관광 카테고리를 기준으로 후보를 필터링한다.
6. 추천 가능한 관광지를 지도와 퀘스트 카드로 표시한다.
7. 사용자가 관광 카테고리, 위치 권역, 탐색 조건을 변경하면 기존 임시 캐시를 폐기하고 OpenAPI를 다시 호출한다.

### 3.2 퀘스트 수행 흐름

1. 사용자가 추천 관광지 또는 퀘스트 목록에서 수행할 퀘스트를 선택한다.
2. 서비스는 퀘스트 유형, 인증 방식, 예상 보상 XP, 연결 뱃지를 보여준다.
3. 사용자가 현장에 도착하면 GPS 반경 조건을 먼저 확인한다.
4. 퀘스트 유형에 따라 사진, 시간대, 이동 거리, 소비 인증 같은 추가 조건을 확인한다.
5. 인증이 통과되면 퀘스트 완료 기록을 저장한다.
6. 완료 XP를 사용자 레벨과 카테고리별 뱃지 진행도에 반영한다.
7. 뱃지 조건을 충족하면 뱃지 또는 뱃지 단계를 갱신한다.
8. 완료 결과를 모험가의 수첩에 기록한다.

### 3.3 수첩 기록 흐름

1. 사용자가 완료한 퀘스트가 생기면 장소, 시간, 카테고리, 획득 XP, 뱃지 정보를 기록한다.
2. 수첩 화면은 완료한 퀘스트와 획득 뱃지를 타임라인 또는 카드 형태로 표시한다.
3. 향후 확장에서는 사용자가 수첩 카드를 이미지로 저장하거나 SNS에 공유할 수 있게 한다.

## 4. 목표 기능 설계

### 4.1 위치 기반 관광지 추천

추천 기능은 현재 위치와 한국관광공사 OpenAPI 조회 결과를 기반으로 한다. 서버는 사용자의 좌표와 필터 조건을 받아 OpenAPI를 호출하고, 응답받은 관광지 후보를 유저 단위 임시 캐시에 저장한다.

추천 점수는 다음 요소를 합산해 계산한다.

- 현재 위치와 관광지 사이의 거리
- 사용자가 입력한 관광 선호도
- 사용자가 이미 완료한 퀘스트 카테고리
- 아직 획득하지 않은 뱃지 또는 다음 단계 뱃지
- 관광지 카테고리와 퀘스트 템플릿의 매칭 가능성
- 현재 시간대 또는 특정 퀘스트 수행 가능 조건

초기 MVP 확장에서는 복잡한 머신러닝 추천보다 규칙 기반 점수화를 사용한다. 데이터가 충분히 쌓인 뒤에는 사용자 행동 이력을 활용한 추천 고도화를 검토한다.

### 4.2 퀘스트 생성

퀘스트 생성은 안정성과 제출 가능성을 위해 템플릿 기반 생성을 1순위로 둔다.

#### 4.2.1 템플릿 기반 생성

관광지 카테고리별로 사전 제작한 템플릿을 준비한다.

- 음식점 및 카페: 소비 인증, 사진 인증, 지역 상권 방문 퀘스트
- 산책로 및 공원: GPS 방문, 이동 거리, 자연 관찰 기록 퀘스트
- 박물관 및 전시관: 전시 관람, 키워드 기록, 사진 인증 퀘스트
- 전망대 및 야경 명소: 시간대 조건, 사진 인증, 방문 기록 퀘스트
- 교통 및 이동형 장소: 이동 거리, 경로 완주, 다음 장소 연결 퀘스트

서버는 OpenAPI 관광지 정보의 카테고리, 장소명, 좌표, 설명을 템플릿에 삽입해 퀘스트 후보를 만든다. 생성된 후보는 금칙 조건, 수행 가능성, 보상 XP 범위, 인증 방식이 올바른지 점검한 뒤 사용자에게 제공한다.

#### 4.2.2 Gemini 기반 생성

Gemini는 퀘스트 문구 다양화와 신규 퀘스트 후보 생성을 위한 보조 수단으로 사용한다. AI 생성 결과는 그대로 사용자에게 제공하지 않고 다음 검증을 통과해야 한다.

- 관광지 정보와 무관한 내용 제거
- 위험하거나 불법적인 행동을 요구하는 문구 제거
- 사유지 침입, 과도한 소비 유도, 안전 위험 행동 제한
- 카테고리, 난이도, 예상 소요 시간, 인증 방식 검증
- XP 보상이 사전 정책 범위 안에 있는지 확인

### 4.3 퀘스트 재사용

다른 사용자를 위해 생성되었고 검증을 통과한 퀘스트는 내부 퀘스트 자산으로 저장해 재사용한다. 단, 한국관광공사 OpenAPI에서 받은 관광지 원본 데이터 전체를 저장하는 방식이 아니라, 서비스가 생성하고 검증한 퀘스트 정의와 필요한 최소 참조 정보만 저장한다.

재사용 대상 데이터는 다음으로 제한한다.

- 퀘스트 제목과 설명
- 퀘스트 유형
- 인증 방식
- 보상 XP
- 연결 뱃지 카테고리
- 관광지 식별을 위한 최소 참조값
- 생성 및 검증 상태

### 4.4 퀘스트 인증

퀘스트 인증은 유형별로 다르게 처리한다.

- 방문형: 관광지 좌표 기준 10m에서 50m 반경 내 GPS 도달 여부 확인
- 이동형: 출발지와 도착지, 이동 거리, 경로 일부를 확인
- 활동형: 특정 시간대, 사진, 체크리스트 수행 여부 확인
- 소비형: 사진 또는 영수증 기반 인증을 검토하되, 개인정보 노출을 최소화
- 테마형: 전시 관람, 키워드 기록, 여러 장소 방문 조합 확인

MVP 확장 1단계에서는 GPS 인증을 우선 구현하고, 사진 인증은 사용자 업로드 파일을 서버에 저장하기 전에 민감 정보와 위치 정보 노출 위험을 검토한다.

### 4.5 뱃지 및 레벨 시스템

뱃지는 퀘스트 카테고리별로 사전 정의한다. 사용자가 퀘스트를 완료하면 해당 카테고리의 뱃지 진행도에 XP가 누적된다. 기준 점수를 충족하면 뱃지를 획득하거나 다음 단계로 업그레이드한다.

예시 뱃지 체계는 다음과 같다.

- 자연 관찰: 초록 탐험가
- 과학 문화: 과학 탐험가
- 원도심 걷기: 대전 워커
- 지역 상권: 빵지순례자
- 이동형: 타슈 라이더
- 야경 기록: 전망 수집가

레벨은 모든 퀘스트 완료 XP를 합산해 계산한다. 레벨은 사용자 전체 활동량을 나타내고, 뱃지는 사용자의 관광 성향을 나타내는 태그로 활용한다.

### 4.6 모험가의 수첩

모험가의 수첩은 사용자의 관광 경험을 축적하는 개인 기록장이다. 완료한 퀘스트, 방문 장소, 획득 뱃지, 이동 거리, 주요 사진 또는 메모를 시간순으로 보여준다.

수첩 기능의 역할은 다음과 같다.

- 사용자가 자신의 관광 기록을 다시 볼 수 있게 한다.
- 뱃지와 레벨 성장의 근거를 보여준다.
- 이후 추천 시스템에서 사용자의 선호도와 활동 이력으로 활용한다.
- 공유 가능한 이미지 카드로 확장해 서비스 홍보 효과를 만든다.

## 5. 시스템 아키텍처

### 5.1 목표 아키텍처

향후 확장 시에는 모바일 웹/PWA, 웹 서버, 앱 서버, 데이터베이스, 외부 API 연동 계층을 분리한다.

```text
모바일 웹/PWA
  ├─ 사용자 위치 및 선호도 입력
  ├─ 추천 관광지와 퀘스트 표시
  ├─ 퀘스트 인증 요청
  └─ 수첩 및 뱃지 화면 표시

웹 서버
  ├─ 정적 파일 제공
  ├─ HTTPS 보안 헤더와 압축 설정
  ├─ 사용자 웹, 파트너 웹, 관리자 웹 라우팅
  └─ API 요청을 앱 서버로 프록시

앱 서버
  ├─ 사용자와 선호도 관리
  ├─ 한국관광공사 OpenAPI 호출
  ├─ 유저 단위 임시 캐시 관리
  ├─ 추천 점수 계산
  ├─ 퀘스트 생성 및 검증
  ├─ 퀘스트 완료 처리
  ├─ 뱃지와 레벨 갱신
  ├─ 수첩 기록 조회
  └─ 리워드 발급과 쿠폰 사용 처리

데이터베이스
  ├─ 사용자 프로필
  ├─ 선호도
  ├─ 완료 퀘스트
  ├─ 뱃지와 레벨 진행도
  ├─ 수첩 기록
  ├─ 검증 완료 퀘스트 자산
  └─ 파트너 상점과 리워드 사용 내역

외부 서비스
  ├─ 한국관광공사 OpenAPI
  ├─ NAVER Maps API
  └─ Gemini API
```

### 5.2 인프라 설계

제출용 운영 기준은 네이버 클라우드 플랫폼 사용을 전제로 한다.

- 프론트엔드: 정적 웹 파일 또는 PWA 배포
- 웹 서버: 정적 파일 제공, HTTPS 보안 헤더, API 프록시
- 앱 서버: Python 앱 서버
- 데이터베이스: 소규모 관리형 관계형 데이터베이스
- 캐시: OpenAPI 응답 최소 필드를 위한 임시 캐시
- 외부 API: 한국관광공사 OpenAPI, NAVER Maps API, Gemini API
- 예상 규모: 전체 사용자 1,000명 이하, 동시 접속 50명 이하

초기 트래픽만 보면 단일 서버와 단일 관계형 데이터베이스 구성이 가능하지만, 운영 구조를 명확히 하기 위해 웹 서버와 앱 서버는 처음부터 역할을 분리한다. 서버 대수는 각각 1대로 시작할 수 있으며, 트래픽이 늘어나면 웹 서버와 앱 서버를 독립적으로 증설한다.

네트워크 구조를 설계하는 단계에서는 PostgreSQL과 MySQL 중 어느 엔진을 사용할지보다 데이터베이스를 어느 네트워크 영역에 둘지가 더 중요하다. 데이터베이스 엔진은 `Private Data Subnet` 안의 관리형 관계형 데이터베이스로 추상화하고, 실제 엔진은 스키마 복잡도, JSON 처리, 위치 기반 확장성, 팀 숙련도에 따라 후속 단계에서 결정한다.

### 5.3 클라우드 네트워크 구조

초기 네트워크는 NCP VPC 안에서 외부 공개 영역, 웹 서버 영역, 앱 서버 영역, 데이터 영역을 분리한다. 외부 사용자는 로드 밸런서를 통해서만 서비스에 접근하고, 웹 서버는 앱 서버로만 API 요청을 전달한다. 앱 서버와 데이터베이스 및 캐시는 인터넷에서 직접 접근할 수 없게 둔다.

```text
사용자 모바일 웹/PWA
  │
  ▼
Global DNS
  │
  ▼
CDN 또는 WAF
  │
  ▼
Public Application Load Balancer
  │
  ▼
Private Web Subnet
  ├─ 웹 서버 1대
  └─ 웹 서버 2대째는 확장 시 추가
        │
        ▼
Internal Application Load Balancer 또는 내부 라우팅
        │
        ▼
Private App Subnet
  ├─ Python 앱 서버 1대
  └─ Python 앱 서버 2대째는 확장 시 추가
        │
        ├─ Private Data Subnet
        │    ├─ 관리형 관계형 데이터베이스
        │    └─ Redis 또는 서버 캐시
        │
        └─ NAT Gateway Subnet
             └─ 한국관광공사 OpenAPI, NAVER Maps REST API, Gemini API 호출
```

초기 VPC 서브넷 예시는 다음과 같다.

| 영역 | 역할 | 예시 CIDR | 외부 직접 접근 |
| --- | --- | --- | --- |
| Load Balancer Subnet | HTTPS 진입점 | `10.30.0.0/24` | 허용 |
| Web Subnet | 정적 파일 제공, API 프록시 | `10.30.10.0/24` | 차단 |
| App Subnet | Python 앱 서버 | `10.30.20.0/24` | 차단 |
| Data Subnet | 관계형 DB와 캐시 | `10.30.30.0/24` | 차단 |
| NAT Gateway Subnet | 내부 서버의 외부 API 호출 출구 | `10.30.250.0/24` | 제한 |

### 5.4 Subnet과 NAT Gateway 역할

Subnet은 VPC 내부를 역할별로 나누는 네트워크 구역이다. 이 프로젝트에서는 공개 진입점, 애플리케이션 서버, 데이터 저장소, 외부 API 호출 출구를 서로 다른 subnet으로 나누어 보안 규칙을 단순하게 만든다.

- Load Balancer Subnet은 사용자가 접근하는 유일한 공개 진입점이다.
- Web Subnet은 정적 파일 제공, 보안 헤더 설정, `/api` 프록시를 담당하는 웹 서버 영역이다.
- App Subnet은 Python 앱 서버가 위치하는 내부 영역이다.
- Data Subnet은 관계형 데이터베이스와 캐시가 위치하는 내부 영역이다.
- NAT Gateway Subnet은 App Subnet의 서버가 외부 API를 호출할 때 사용하는 출구이다.

NAT Gateway는 private subnet의 서버가 인터넷으로 나가는 요청을 할 수 있게 해준다. 예를 들어 앱 서버는 한국관광공사 OpenAPI, NAVER Maps REST API, Gemini API를 호출해야 하지만, 외부 사용자가 앱 서버에 직접 접근하면 안 된다. NAT Gateway를 사용하면 내부 서버의 외부 요청은 가능하게 하면서 외부에서 내부 서버로 직접 들어오는 경로는 만들지 않을 수 있다.

웹 서버와 앱 서버를 분리하면 정적 파일 제공, TLS 관련 설정, 보안 헤더, 압축, API 프록시, rate limit 같은 웹 계층 책임과 추천, 퀘스트, 인증, 리워드 같은 비즈니스 로직 책임을 분리할 수 있다. 약간의 서버 비용과 운영 지점이 늘어나지만, 지역 상권 연계와 파트너 포털이 추가될 때 사용자 웹, 파트너 웹, 내부 API를 같은 앱 서버에 직접 노출하지 않고 웹 계층에서 경로와 접근 정책을 통제할 수 있다.

### 5.5 지역 상권 연계 확장 인프라

지역 상권과 연계하는 단계에서는 기존 사용자 서비스 외에 파트너 상점이 쿠폰과 리워드 사용을 확인할 수 있는 관리 화면이 필요하다. 이 기능은 별도 시스템으로 완전히 분리하기보다 같은 VPC 안에서 웹 계층과 앱 계층의 역할을 확장한다.

```text
일반 사용자
  └─ questbook.example.com
       └─ 웹 서버
            └─ 앱 서버: 추천, 퀘스트, 뱃지, 수첩, 리워드 발급

지역 상점 담당자
  └─ partner.questbook.example.com
       └─ 웹 서버
            └─ 앱 서버: 파트너 로그인, 쿠폰 조회, 사용 처리

운영 관리자
  └─ admin.questbook.example.com
       └─ 웹 서버
            └─ 앱 서버: 파트너 승인, 리워드 정책 관리, 정산용 내역 조회
```

상권 연계 단계에서 추가로 고려할 인프라 구성은 다음과 같다.

- 파트너 포털은 같은 Web Subnet에 배포하되 도메인과 라우팅을 분리한다.
- 파트너 API와 관리자 API는 같은 App Subnet에서 서비스하되 권한과 인증 정책을 분리한다.
- 쿠폰 이미지, 상점 로고, 공유 이미지 카드는 Object Storage에 저장하고 CDN으로 제공한다.
- 쿠폰 사용 처리, QR 코드 검증, 중복 사용 방지는 관계형 DB 트랜잭션으로 처리한다.
- 사용 완료 알림이나 쿠폰 만료 알림이 필요해지면 메시지 큐 또는 작업 큐를 추가한다.
- 파트너 상점에는 사용자 개인정보 전체를 제공하지 않고 쿠폰 코드, 사용 가능 여부, 사용 시각 등 최소 정보만 보여준다.

### 5.6 캐시 배치 원칙

캐시는 영구 데이터베이스가 아니다. 이 프로젝트의 캐시는 브라우저 캐시, 서버 캐시, 영구 데이터베이스를 구분해서 사용한다.

| 위치 | 저장 대상 | 목적 |
| --- | --- | --- |
| 브라우저 캐시 | 정적 파일, 최근 화면 상태, 마지막 추천 결과 | UX 개선과 재표시 |
| 서버 캐시 | OpenAPI 응답에서 추출한 최소 관광지 후보 필드 | 추천 계산과 퀘스트 생성 재사용 |
| 관계형 데이터베이스 | 사용자, 선호도, 뱃지, 레벨, 수첩, 공용 퀘스트 | 서비스 영구 상태 저장 |

한국관광공사 OpenAPI 응답은 사용자의 기기에도 일시적으로 표시될 수 있지만, 추천 점수 계산과 퀘스트 생성의 기준 데이터는 서버가 관리해야 한다. 클라이언트 데이터는 조작될 수 있고, OpenAPI Key도 서버에서 보호해야 하기 때문이다. 따라서 브라우저 캐시는 UX 보조 용도로만 사용하고, 서버 캐시는 추천 및 생성 로직을 위한 임시 저장소로 사용한다.

초기 MVP에서는 앱 서버 메모리 캐시로 시작할 수 있다. 앱 서버가 2대 이상으로 늘거나 캐시 공유가 필요해지면 Redis 또는 NCP의 관리형 캐시 서비스로 분리한다. OpenAPI 캐시는 유저, 위치 권역, 관광 카테고리, 조회 시점을 기준으로 묶고 기본 TTL은 30분으로 둔다.

### 5.7 데이터베이스 분리 원칙

초기에는 사용자 데이터와 퀘스트 데이터를 물리적으로 다른 데이터베이스 서버에 나누지 않는다. 하나의 관계형 데이터베이스 안에서 schema 또는 table prefix로 도메인을 나누는 방식을 우선한다.

```text
auth
  - 로그인 계정
  - 비밀번호 해시

user
  - 사용자 프로필
  - 선호도

quest
  - 퀘스트 템플릿
  - 검증 완료 공용 재사용 퀘스트
  - 사용자별 추천 및 수행 상태

reward
  - 레벨 정책
  - 뱃지 정의
  - 사용자 뱃지 진행도
  - 지역 상권 리워드와 쿠폰

note
  - 모험가의 수첩 기록

partner
  - 지역 상점 정보
  - 파트너 담당자 계정
  - 쿠폰 사용 처리 내역
```

사용자가 퀘스트를 완료하면 완료 기록, XP, 레벨, 뱃지 진행도, 수첩 기록이 함께 갱신된다. 이 흐름은 하나의 트랜잭션으로 처리하는 것이 안전하다. 데이터베이스를 처음부터 유저 DB와 퀘스트 DB로 물리 분리하면 cross-database transaction, 백업, migration, 장애 대응이 복잡해진다.

물리적 분리는 다음 조건이 생겼을 때 검토한다.

- 유저 개인정보를 국가 또는 지역별로 분리해야 하는 경우
- 공용 퀘스트 라이브러리의 읽기 트래픽이 관계형 DB를 압박하는 경우
- 분석 로그와 운영 데이터의 저장 주기와 조회 패턴이 크게 달라지는 경우
- 인증, 퀘스트, 리워드 도메인을 서로 다른 팀이 독립 운영해야 하는 경우

따라서 현재 단계의 결정은 다음과 같다.

- 네트워크 설계에서는 PostgreSQL 또는 MySQL 중 하나를 고정하지 않는다.
- 관계형 DB는 private data subnet에 배치한다.
- 초기에는 하나의 관리형 관계형 DB를 사용한다.
- 도메인 분리는 schema 또는 table 구조로 처리한다.
- OpenAPI 응답은 영구 DB가 아니라 서버 캐시에만 임시 저장한다.

## 6. 데이터 흐름과 저장 정책

### 6.1 한국관광공사 OpenAPI 사용 원칙

한국관광공사 OpenAPI는 이 프로젝트의 필수 데이터 원천이다. 관광지 원본 데이터는 서버 DB에 영구 저장하지 않는다. 사용자가 현재 관광 중 불러온 OpenAPI 응답은 유저 단위 임시 캐시로만 활용한다.

임시 캐시 정책은 다음과 같다.

- 캐시 범위: 특정 사용자, 현재 위치 권역, 관광 카테고리, 조회 시점에 묶인다.
- 캐시 목적: 같은 관광 흐름 안에서 다음 관광지 추천과 퀘스트 후보 생성에 재사용한다.
- 폐기 조건: 기본 유효 시간이 지나면 폐기한다. 초기 설계 기본값은 30분이다.
- 재조회 조건: 사용자가 지역을 크게 이동하거나, 관광 카테고리를 변경하거나, 명시적으로 새로고침하면 OpenAPI를 다시 호출한다.
- 저장 금지: 관광지 원본 응답 전체를 서비스 DB에 장기 저장하지 않는다.

이 정책은 OpenAPI를 필수로 사용하면서도, 사용자가 방금 조회한 관광지 후보를 다음 추천에 활용할 수 있게 하기 위한 절충안이다.

### 6.2 영구 저장 대상

서비스 DB에 저장하는 데이터는 사용자 서비스 경험을 구성하는 내부 데이터로 제한한다.

- 사용자 프로필
- 사용자 선호도
- 완료한 퀘스트 이력
- 획득 뱃지와 뱃지 단계
- 레벨과 XP 진행도
- 모험가의 수첩 기록
- 생성 후 검증된 퀘스트 정의
- 퀘스트 추천과 완료 처리를 위한 최소 참조 정보
- 지역 상점 파트너 정보
- 리워드와 쿠폰 발급 및 사용 처리 내역

### 6.3 저장하지 않는 대상

다음 데이터는 원칙적으로 영구 저장하지 않는다.

- 한국관광공사 OpenAPI 원본 응답 전체
- 사용자의 실시간 위치 이동 경로 전체
- 사진 인증 원본에 포함된 불필요한 개인정보
- API Key, access token, secret 값
- 인증에 필요하지 않은 영수증 상세 정보

## 7. 주요 데이터 모델

아래 모델은 설계 기준이며, 실제 구현 시 데이터베이스 스키마와 API 응답 형식에 맞게 조정한다.

### 7.1 User

사용자 기본 정보를 나타낸다.

- `id`: 사용자 식별자
- `nickname`: 화면에 표시할 이름
- `level`: 현재 레벨
- `totalXp`: 누적 XP
- `createdAt`: 가입 시각
- `lastActiveAt`: 마지막 활동 시각

### 7.2 Preference

사용자의 관광 선호도를 나타낸다.

- `userId`: 사용자 식별자
- `categories`: 선호 관광 카테고리 목록
- `distanceRange`: 추천 거리 범위
- `pace`: 선호 활동 강도
- `updatedAt`: 선호도 갱신 시각

### 7.3 TourPlaceCache

한국관광공사 OpenAPI 응답을 유저 단위로 임시 보관하는 캐시이다.

- `userId`: 사용자 식별자
- `regionKey`: 조회 지역 또는 좌표 권역
- `categoryKey`: 관광 카테고리
- `fetchedAt`: OpenAPI 호출 시각
- `expiresAt`: 캐시 만료 시각
- `places`: 추천 계산에 필요한 관광지 후보 목록

`TourPlaceCache`는 영구 관광지 DB가 아니라 현재 관광 흐름 안에서만 유지되는 임시 데이터이다.

### 7.4 ReusableQuest

검증을 통과해 모든 사용자에게 재사용할 수 있는 공용 퀘스트 자산이다. 특정 사용자의 과거 퀘스트 목록이 아니라, 어떤 사용자 요청 과정에서 생성되었더라도 이후 다른 사용자에게 추천할 수 있는 내부 퀘스트 라이브러리로 관리한다.

- `id`: 공용 퀘스트 식별자
- `title`: 퀘스트 제목
- `description`: 퀘스트 설명
- `type`: 방문형, 이동형, 활동형, 소비형, 테마형 중 하나
- `category`: 뱃지와 추천에 사용할 카테고리
- `rewardXp`: 완료 보상 XP
- `verificationType`: GPS, 사진, 시간, 이동 거리, 소비 인증 등
- `placeReference`: 관광지 최소 참조 정보
- `source`: 템플릿, Gemini 중 하나
- `reviewStatus`: 생성 결과 검증 상태
- `createdForUserId`: 최초 생성 요청과 연결된 사용자 식별자
- `isReusable`: 다른 사용자에게 추천 가능한지 여부
- `reuseCount`: 추천 또는 할당에 재사용된 횟수
- `completionCount`: 완료된 횟수

`placeReference`에는 한국관광공사 OpenAPI 원본 응답 전체를 저장하지 않는다. 퀘스트 재사용과 인증에 필요한 최소 참조값만 저장한다.

### 7.5 UserQuestInstance

공용 퀘스트가 특정 사용자에게 추천되거나 사용자가 수행하기로 선택했을 때 생성되는 사용자별 퀘스트 상태이다.

- `id`: 사용자별 퀘스트 인스턴스 식별자
- `userId`: 사용자 식별자
- `reusableQuestId`: 공용 퀘스트 식별자
- `status`: 추천됨, 수락됨, 진행 중, 완료됨, 포기됨 중 하나
- `recommendedAt`: 추천 시각
- `acceptedAt`: 사용자가 수행하기로 선택한 시각
- `expiresAt`: 추천 또는 수행 가능 만료 시각
- `completedAt`: 완료 시각

이 모델을 분리하면 같은 공용 퀘스트를 여러 사용자가 수행하더라도 각 사용자의 진행 상태와 완료 여부를 독립적으로 관리할 수 있다.

### 7.6 QuestCompletion

사용자가 퀘스트를 완료한 기록이다.

- `id`: 완료 기록 식별자
- `userId`: 사용자 식별자
- `userQuestInstanceId`: 사용자별 퀘스트 인스턴스 식별자
- `reusableQuestId`: 공용 퀘스트 식별자
- `completedAt`: 완료 시각
- `earnedXp`: 획득 XP
- `verificationResult`: 인증 결과
- `noteId`: 생성된 수첩 기록 식별자

### 7.7 Badge

뱃지 정의와 사용자 획득 상태를 나타낸다.

- `id`: 뱃지 식별자
- `category`: 연결 퀘스트 카테고리
- `name`: 뱃지 이름
- `level`: 뱃지 단계
- `requiredXp`: 획득 또는 다음 단계에 필요한 XP
- `userProgressXp`: 사용자별 진행 XP
- `earnedAt`: 획득 시각

### 7.8 LevelProgress

사용자 레벨 진행도를 나타낸다.

- `userId`: 사용자 식별자
- `currentLevel`: 현재 레벨
- `currentXp`: 현재 레벨 내 XP
- `totalXp`: 누적 XP
- `nextLevelRequiredXp`: 다음 레벨까지 필요한 XP

### 7.9 AdventureNote

모험가의 수첩에 표시할 사용자 기록이다.

- `id`: 수첩 기록 식별자
- `userId`: 사용자 식별자
- `reusableQuestId`: 연결된 공용 퀘스트 식별자
- `questCompletionId`: 연결된 완료 기록 식별자
- `placeName`: 표시용 장소명
- `summary`: 활동 요약
- `badges`: 연결된 뱃지 목록
- `distanceKm`: 기록된 이동 거리
- `createdAt`: 기록 생성 시각
- `shareImageUrl`: 향후 공유 이미지 경로

### 7.10 PartnerStore

지역 상권 연계 단계에서 쿠폰이나 리워드를 제공하는 상점 또는 기관 정보이다.

- `id`: 파트너 상점 식별자
- `name`: 상점 또는 기관 이름
- `category`: 음식점, 카페, 전통시장, 체험 공간, 기념품점 등 분류
- `area`: 대전 내 권역 또는 행정동
- `address`: 표시용 주소
- `contactPhone`: 운영 연락처
- `status`: 신청됨, 승인됨, 중지됨 중 하나
- `createdAt`: 등록 시각
- `updatedAt`: 갱신 시각

파트너 상점 정보는 OpenAPI 관광지 원본을 복제하는 용도가 아니라, 서비스가 직접 제휴한 상점과 리워드 제공 조건을 관리하기 위한 내부 운영 데이터이다.

### 7.11 PartnerAccount

지역 상점 담당자가 파트너 포털에서 쿠폰 사용 여부를 확인하고 사용 처리할 수 있게 하는 계정이다.

- `id`: 파트너 계정 식별자
- `partnerStoreId`: 소속 파트너 상점 식별자
- `loginId`: 로그인 ID
- `passwordHash`: 비밀번호 해시
- `role`: 소유자, 직원, 조회 전용 중 하나
- `status`: 활성, 잠김, 비활성 중 하나
- `lastLoginAt`: 마지막 로그인 시각
- `createdAt`: 생성 시각

파트너 계정도 사용자 계정과 마찬가지로 비밀번호 원문을 저장하지 않는다.

### 7.12 RewardOffer

사용자에게 발급할 수 있는 쿠폰, 할인권, 기념품 교환권, 체험권 같은 리워드 정의이다.

- `id`: 리워드 식별자
- `partnerStoreId`: 제공 파트너 상점 식별자
- `title`: 리워드 제목
- `description`: 리워드 설명
- `rewardType`: 할인 쿠폰, 무료 제공, 기념품, 체험권 등 유형
- `issueCondition`: 발급 조건
- `validFrom`: 사용 가능 시작 시각
- `validUntil`: 사용 가능 종료 시각
- `totalQuantity`: 총 발급 가능 수량
- `issuedQuantity`: 발급된 수량
- `status`: 준비 중, 활성, 소진, 종료, 중지 중 하나

발급 조건은 특정 퀘스트 완료, 특정 뱃지 획득, 특정 카테고리 XP 달성, 기간 한정 이벤트 등으로 정의할 수 있다.

### 7.13 UserReward

특정 사용자에게 실제로 발급된 리워드 또는 쿠폰이다.

- `id`: 사용자 리워드 식별자
- `userId`: 사용자 식별자
- `rewardOfferId`: 리워드 정의 식별자
- `status`: 발급됨, 사용됨, 만료됨, 취소됨 중 하나
- `issueReason`: 발급 사유
- `issuedAt`: 발급 시각
- `expiresAt`: 만료 시각
- `redeemedAt`: 사용 처리 시각
- `redemptionCodeHash`: 쿠폰 코드 해시

쿠폰 코드는 중복 사용 방지를 위해 서버에서 생성하고, DB에는 원문 대신 검증 가능한 해시 또는 별도 보호된 값을 저장한다. 사용자가 화면에서 제시하는 QR 코드나 숫자 코드는 `RewardRedemption` 처리 시 검증한다.

### 7.14 RewardRedemption

파트너 상점에서 쿠폰 또는 리워드를 실제로 사용 처리한 기록이다.

- `id`: 사용 처리 기록 식별자
- `userRewardId`: 사용자 리워드 식별자
- `partnerStoreId`: 사용 처리 상점 식별자
- `partnerAccountId`: 사용 처리 담당자 계정 식별자
- `redeemedAt`: 사용 처리 시각
- `status`: 성공, 실패, 취소 중 하나
- `failureReason`: 실패 사유
- `requestMetadata`: 사용 처리 요청의 최소 진단 정보

리워드 사용 처리는 중복 사용이 발생하지 않도록 관계형 DB 트랜잭션 안에서 `UserReward` 상태 변경과 `RewardRedemption` 생성을 함께 처리한다.

## 8. API 흐름 설계

### 8.1 추천 API 흐름

1. 클라이언트가 현재 위치, 관광 카테고리, 선호 조건을 앱 서버에 전달한다.
2. 앱 서버는 같은 사용자, 같은 지역, 같은 카테고리의 유효한 `TourPlaceCache`가 있는지 확인한다.
3. 유효한 캐시가 있으면 캐시를 사용한다.
4. 캐시가 없거나 만료되었거나 조건이 바뀌었으면 한국관광공사 OpenAPI를 호출한다.
5. OpenAPI 응답에서 추천 계산에 필요한 필드만 추출해 유저 단위 임시 캐시에 저장한다.
6. 앱 서버는 사용자 이력과 선호도를 반영해 추천 점수를 계산한다.
7. 추천 관광지와 연결 가능한 퀘스트 후보를 클라이언트에 반환한다.

### 8.2 퀘스트 생성 API 흐름

1. 추천 관광지 후보를 입력으로 받는다.
2. 관광지 카테고리에 맞는 퀘스트 템플릿을 선택한다.
3. 장소명, 위치, 카테고리, 설명을 템플릿에 삽입한다.
4. 필요하면 Gemini를 호출해 문구 다양화 또는 추가 후보를 생성한다.
5. 생성 결과를 검증한다.
6. 검증된 퀘스트를 `ReusableQuest`로 저장해 모든 사용자가 활용 가능한 공용 퀘스트 자산으로 만든다.
7. 해당 퀘스트가 특정 사용자에게 추천되거나 선택되면 `UserQuestInstance`를 생성한다.

### 8.3 퀘스트 완료 API 흐름

1. 클라이언트가 퀘스트 완료 요청과 인증 정보를 제출한다.
2. 앱 서버는 인증 방식에 따라 GPS, 시간, 사진, 이동 거리 조건을 확인한다.
3. 인증이 실패하면 실패 사유와 재시도 가능 여부를 반환한다.
4. 인증이 성공하면 `QuestCompletion`을 생성한다.
5. 사용자 XP와 `LevelProgress`를 갱신한다.
6. 연결 카테고리의 `Badge` 진행도를 갱신한다.
7. 필요한 경우 새 뱃지 또는 뱃지 단계를 부여한다.
8. 연결된 `ReusableQuest`의 완료 횟수와 재사용 통계를 갱신한다.
9. `AdventureNote`를 생성한다.

### 8.4 리워드 발급 API 흐름

1. 퀘스트 완료, 뱃지 획득, 레벨 상승, 이벤트 참여 같은 리워드 발급 트리거가 발생한다.
2. 앱 서버는 활성 상태의 `RewardOffer` 중 발급 조건을 만족하는 항목을 조회한다.
3. 발급 수량, 유효 기간, 사용자별 중복 발급 제한을 확인한다.
4. 조건을 만족하면 `UserReward`를 생성하고 쿠폰 코드 또는 QR 코드 검증값을 발급한다.
5. `RewardOffer.issuedQuantity`를 갱신한다.
6. 사용자 앱은 발급된 리워드 목록과 만료 시각을 표시한다.

### 8.5 리워드 사용 처리 API 흐름

1. 사용자가 지역 상점에서 앱의 쿠폰 QR 코드 또는 사용 코드를 제시한다.
2. 파트너 담당자는 파트너 포털에서 QR 코드 또는 사용 코드를 스캔하거나 입력한다.
3. 앱 서버는 `PartnerAccount` 권한과 `PartnerStore` 상태를 확인한다.
4. `UserReward`가 해당 상점에서 사용 가능한지, 만료되지 않았는지, 이미 사용되지 않았는지 검증한다.
5. 검증이 통과하면 하나의 트랜잭션 안에서 `UserReward` 상태를 사용됨으로 변경하고 `RewardRedemption`을 생성한다.
6. 검증이 실패하면 실패 사유만 반환하고 사용자 개인정보는 파트너 화면에 노출하지 않는다.
7. 운영 관리자는 관리자 화면에서 파트너별 발급 수량, 사용 수량, 실패 사유, 기간별 사용 내역을 조회한다.

## 9. 보안 및 개인정보 설계

### 9.1 API Key 관리

NAVER Maps REST API Key, 한국관광공사 OpenAPI Key, Gemini API Key는 서버 환경변수 또는 배포 환경의 secret 관리 기능에 저장한다. 브라우저에는 공개가 필요한 Key ID 또는 클라이언트용 설정만 전달한다.

NAVER Geocoding과 Reverse Geocoding은 서버 프록시를 통해 호출한다. 이 구조를 유지해 브라우저에 REST API Key가 노출되지 않도록 한다.

### 9.2 위치 정보 보호

사용자 위치는 추천과 인증에 필요한 순간에만 사용한다. 실시간 이동 경로 전체를 기본 저장하지 않으며, 퀘스트 완료에 필요한 최소 좌표 또는 인증 결과만 저장한다.

### 9.3 사진 및 소비 인증 보호

사진 인증과 영수증 인증은 개인정보 노출 가능성이 있다. 향후 구현 시 다음 원칙을 적용한다.

- 인증에 필요한 최소 파일만 업로드한다.
- 영수증의 결제 수단, 승인번호, 개인 식별 정보는 저장하지 않는다.
- 사진 원본의 위치 메타데이터 저장 여부를 검토하고 필요 시 제거한다.
- 제출용 MVP에서는 민감한 인증 데이터 저장 없이 인증 흐름을 우선 시연한다.

### 9.4 파트너 포털과 리워드 보안

지역 상점 파트너는 쿠폰 사용 처리에 필요한 최소 정보만 확인할 수 있어야 한다. 파트너 포털에는 사용자의 전체 프로필, 위치 이력, 퀘스트 상세 이력, 연락처를 노출하지 않는다.

- 파트너 담당자는 `PartnerAccount`로 별도 로그인한다.
- 파트너 계정은 자신이 소속된 `PartnerStore`의 리워드만 조회하고 처리할 수 있다.
- 쿠폰 원문 코드는 DB에 그대로 저장하지 않고 해시 또는 별도 보호된 검증값으로 저장한다.
- 쿠폰 사용 처리는 중복 사용을 막기 위해 관계형 DB 트랜잭션으로 처리한다.
- 관리자 화면은 파트너 승인, 리워드 정책 관리, 사용 내역 조회 권한을 일반 파트너 계정과 분리한다.
- 리워드 사용 로그에는 쿠폰 검증 결과와 진단 정보만 남기고 불필요한 사용자 개인정보는 남기지 않는다.

## 10. 운영 및 장애 대응

### 10.1 외부 API 장애

한국관광공사 OpenAPI 호출이 실패하면 사용자에게 재시도 안내와 기존 화면 유지 상태를 제공한다. 유효한 유저 단위 임시 캐시가 있으면 해당 캐시를 사용하되, 만료된 원본 데이터를 장기 저장소처럼 사용하지 않는다.

NAVER Maps 로드에 실패하면 목업 지도 또는 장소 목록 중심 화면으로 fallback한다.

Gemini 호출이 실패하면 템플릿 기반 퀘스트만 제공한다.

### 10.2 성능 기준

동시 접속 50명 이하를 기준으로 초기 웹 서버와 앱 서버는 각각 단일 인스턴스로 처리 가능하다. 다만 웹 서버와 앱 서버는 역할과 배포 단위를 분리해 두고, 트래픽이 늘어나면 웹 서버와 앱 서버를 독립적으로 증설한다. OpenAPI 호출은 사용자 요청마다 무조건 반복하지 않고 유저 단위 임시 캐시를 사용해 외부 API 호출량을 줄인다.

### 10.3 로그 정책

운영 로그에는 요청 성공 여부, 외부 API 호출 결과, 오류 유형, 응답 시간 등 진단에 필요한 정보만 남긴다. API Key, 원본 인증 파일, 민감한 위치 상세 이력은 로그에 남기지 않는다.

## 11. 향후 개발 단계

### 11.1 1단계: 현재 MVP 정리

## 12. 검수 기준

### 12.1 문서 검수 기준

- `Design.md`의 필수 요구사항과 기능 항목이 모두 반영되어야 한다.
- 한국관광공사 OpenAPI 데이터 정책은 원본 데이터 영구 저장 금지와 유저 단위 임시 캐시 허용으로 일관되어야 한다.
- 현재 구현된 기능과 향후 구현할 기능이 분리되어야 한다.
- 문서는 한국어로 작성되어야 한다.
- API Key, `.env` 값, 개인 인증 정보가 포함되지 않아야 한다.

### 12.2 기능 시나리오 검수 기준

- 사용자가 현재 위치를 기준으로 주변 관광지를 추천받는 흐름이 설명되어야 한다.
- 관광 카테고리 또는 지역이 변경되면 OpenAPI를 다시 호출하는 흐름이 설명되어야 한다.
- 퀘스트 완료 후 XP, 뱃지, 수첩 기록이 갱신되는 흐름이 설명되어야 한다.
- NAVER Maps REST API Key가 서버 프록시에서 보호되는 구조가 설명되어야 한다.
- 웹 서버와 앱 서버의 역할이 분리되어야 한다.
- 지역 상권 연계 시 파트너 포털, 리워드 발급, 쿠폰 사용 처리 흐름이 설명되어야 한다.

## 13. 설계상 결정 사항

- 새 설계 문서는 `PROJECT_DESIGN.md`로 관리한다.
- 기존 `Design.md`는 원본 요구사항 메모로 유지한다.
- 관광지 원본 데이터는 영구 저장하지 않는다.
- OpenAPI 응답은 유저 단위 임시 캐시에 한해 사용한다.
- 임시 캐시 기본 유효 시간은 30분으로 둔다.
- 관광 지역 또는 카테고리 변경 시 캐시를 폐기하고 OpenAPI를 다시 호출한다.
- 브라우저 캐시는 UX 보조 용도로 사용하고, 추천 및 퀘스트 생성 기준 데이터는 서버 캐시에서 관리한다.
- 웹 서버와 앱 서버는 역할을 분리해 설계하고, 초기에는 각각 1대 규모로 시작할 수 있다.
- 네트워크 설계 단계에서는 PostgreSQL과 MySQL 중 특정 DB 엔진을 고정하지 않는다.
- 초기에는 하나의 관리형 관계형 데이터베이스를 사용하고, 사용자 데이터와 퀘스트 데이터는 schema 또는 table 구조로 논리 분리한다.
- 검증 완료된 퀘스트는 개인 과거 목록이 아니라 모든 사용자가 재사용할 수 있는 `ReusableQuest` 공용 자산으로 저장한다.
- 지역 상권 연계는 `PartnerStore`, `PartnerAccount`, `RewardOffer`, `UserReward`, `RewardRedemption`을 중심으로 확장한다.
- 퀘스트 생성은 템플릿 기반을 기본으로 하고, Gemini는 보조 수단으로 사용한다.
- 제출용 운영 규모는 전체 사용자 1,000명 이하, 동시 접속 50명 이하를 상한 기준으로 한다.
